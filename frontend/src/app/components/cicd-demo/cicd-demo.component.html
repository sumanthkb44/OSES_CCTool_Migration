What are Reusable Workflows?

Reusable workflows are GitHub Actions workflows that can be called from other workflows. This promotes the DRY (Don't Repeat Yourself) principle and allows you to:

  • Centralize logic: Define workflow logic once and reuse it across multiple repositories
  • Simplify maintenance: Update the workflow in one place and all callers benefit
  • Ensure consistency: Same process runs everywhere with the same steps
  • Parameterize behavior: Pass inputs to customize workflow execution

Artifact-Based Input Passing

In GitHub Actions, jobs can communicate by uploading and downloading artifacts. This is especially useful for passing build outputs or dependencies to subsequent jobs:

┌────────────────────────┐ │ Job 1: Build │ │ ───────────────── │ │ - Compile code │ │ - Collect deps │ │ - Upload artifact │ └───────┬────────────────┘ │ │ [Artifact: dependencies.zip] │ v ┌────────────────────────┐ │ Job 2: Clear │ │ ───────────────── │ │ - Download artifact │ │ - Scan dependencies │ │ - Generate SBOM │ │ - Upload SBOM │ └───────┬────────────────┘ │ │ [Artifact: sbom.json] │ v ┌────────────────────────┐ │ Job 3: Deploy │ │ ───────────────── │ │ - Download SBOM │ │ - Verify compliance │ │ - Deploy to prod │ └────────────────────────┘

SBOM Generation

SBOM (Software Bill of Materials) is a machine-readable inventory of all components in your software. It includes:

  • Component names: All dependencies (direct and transitive)
  • Versions: Exact version numbers for reproducibility
  • Licenses: License information for compliance
  • Vulnerabilities: Known CVEs associated with components
  • Relationships: Dependency graph showing how components relate

SBOMs are critical for supply chain security, compliance audits, and vulnerability management.

Using the Reusable Workflow

To use the Continuous Clearing reusable workflow in your repository, create a caller workflow:

Step 1: Create the reusable workflow

Save the reusable workflow YAML (from the "Reusable Workflow YAML" tab) to:

.github/workflows/continuous-clearing.yml

Step 2: Create a caller workflow

Create a new workflow file that calls the reusable workflow:

.github/workflows/ci.yml 
name: CI Pipeline with Continuous Clearing

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # Call the reusable Continuous Clearing workflow
  continuous-clearing:
    name: Run Continuous Clearing
    uses: ./.github/workflows/continuous-clearing.yml
    with:
      sbom-format: spdx
      fail-on-critical: true
      working-directory: ./frontend

  # After clearing passes, run other jobs
  build:
    name: Build Application
    needs: continuous-clearing
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: npm run build

Step 3: Customize with inputs

The reusable workflow accepts these inputs:

Input Type Default Description
sbom-format string spdx SBOM format (spdx or cyclonedx)
fail-on-critical boolean true Fail build on critical vulnerabilities
working-directory string . Directory to scan (for monorepos)

Step 4: Access outputs

The workflow outputs the SBOM artifact name, which you can use in subsequent jobs:

jobs:
  clearing:
    uses: ./.github/workflows/continuous-clearing.yml
  
  deploy:
    needs: clearing
    runs-on: ubuntu-latest
    steps:
      - name: Download SBOM
        uses: actions/download-artifact@v4
        with:
          name: ${{ '{{' }} needs.clearing.outputs.sbom-artifact {{ '}}' }}
